# Vega 设计理念

## 项目定位

Vega 是一个 AI 优先的个人知识库。核心价值不是存储，而是**让文档可被 AI 检索和使用**，给"写文档"一个真正能被利用起来的理由。

## 核心理念

### AI 优先，人辅助

主要读写方是 AI，人偶尔介入。数据格式优先考虑 AI 的解析效率，但人类也能直接阅读和编辑。

### 存意图，不存实现

存"为什么这么做"和"用什么方式做"，不存具体代码。清晰的设计描述比代码更有价值——描述写得够清楚，AI 就能从理解中生成相同的代码。

### 跨项目持久记忆

知识超越单个项目的生命周期。项目可以删除，但设计思想、经验教训保留在 Vega 中，后续项目可以参考。

### 易迁移

纯 Markdown 文件，不绑定特定工具或平台。复制文件夹即可完成迁移。

## 设计决策

### 为什么用 Markdown 文件

- 人和 AI 都原生好读好改
- 人偶尔需要直接查看和编辑
- 不需要额外工具就能打开
- 比 SQLite 等数据库更透明、更易维护

### 为什么提供 CLI

CLI 是给 AI 用的接口，不是给人类的。交互模型：

```
人类 ←→ Markdown 文件 ←→ AI（通过 CLI/MCP）
```

两边都能直接接触数据，方式不同。

### 为什么分 projects/ 和 user/

- `projects/` — 硬性规定，AI 存储项目记忆的专属区域
- `user/` — 存放其他所有内容（个人偏好、配置信息等），内部结构灵活，由用户自定义

两大类足够，不再增加更多分类层级。

### 为什么不存具体代码

代码可以从清晰的设计描述中重新生成。存代码反而是冗余，且容易过时。设计思想更稳定、更有参考价值。

## 诞生背景

### 痛点一：项目恢复困难

AI 对话上下文过长时需要压缩或重开对话，重开后又得重新让 AI 理解项目。虽然可以在项目 docs/ 内留文档，但 Vega 给了一个统一的地方来写和检索这类文档。

### 痛点二：跨项目经验难以复用

项目经验越来越多，但旧项目可能已删除，即使还在也只能从源码反推。Vega 提供了一个地方来存储设计思想，让 AI 可以跨项目参考过往经验。

### 痛点三：个人数据分散

个人文档、配置信息散落在电脑各处。Vega 提供一个集中存储，AI 可以快速检索，也能在重装系统后帮助恢复配置。
